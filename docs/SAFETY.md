# PSL Memory Safety Review

**Last refreshed:** 2026-05-10 (post-PQ phases 1-3 merge).
**Audience:** auditor, partner technical due diligence, SDK
integrator evaluating PSL as a settlement-layer dependency.
**Companion docs:** [`UNWRAP_AUDIT.md`](UNWRAP_AUDIT.md) for the
panic-path audit; [`SECURITY_REVIEW.md`](SECURITY_REVIEW.md) for the
full audit-package security review.

## Headline

**Zero `unsafe` blocks on PSL production paths.** PSL is pure-Rust
across all 12 workspace crates (`crypto`, `crypto_agility`,
`consensus`, `sequencer`, `light_client`, `ternary_vm`,
`agent_contracts`, `agent_wallet`, `agent_protocol`, `agent_sdk`,
`pilot/issuer_demo`) with no `unsafe` blocks introduced by PSL
itself.

The only PSL-owned `unsafe` reference is in
`legacy/rust_runner/src/bin/dump_logits.rs:98`, which is a
diagnostic binary in the **frozen-per-ADR-0001** legacy crate. It
performs a `&[T] → &[u8]` view-cast for binary dumping; it is not
on any production execution path and is excluded from the v0.1.0
audit scope.

## Why this matters

Per [`ADR-0001`](decisions/0001-retire-legacy-fp64-runner.md) and
the [`ARCHITECTURE`](ARCHITECTURE.md) trust contract, the
trace_hash on which dispute-by-re-execution rests must produce
bit-exact output across machines. Memory-safety bugs (use-after-
free, buffer overruns, data races) can corrupt output bytes
silently, defeating the determinism property. Pure-safe-Rust
removes the entire class.

## Transitive dependency `unsafe` posture

PSL pulls in ~217 transitive crates (see
[`LICENSE_REVIEW.md`](LICENSE_REVIEW.md)). A subset of these contain
`unsafe` blocks — this is normal and unavoidable for a Rust project
that does any system I/O, async runtime work, or FFI to NIST
reference C code (e.g., `pqcrypto-mldsa`).

Critical-path transitive deps and their `unsafe` posture:

| Dependency             | Has `unsafe`? | Audit status |
| ---                    | ---           | ---          |
| `ed25519-dalek`        | yes (constant-time field ops) | Audited multiple times; canonical Rust ed25519. |
| `curve25519-dalek`     | yes (field arithmetic)        | Same audit lineage as ed25519-dalek. |
| `x25519-dalek`         | yes (field ops)               | Same. |
| `blake3`               | yes (SIMD intrinsics)         | Reference implementation, widely reviewed. |
| `pqcrypto-mldsa`       | yes (C FFI to PQClean)        | NIST-audited reference C implementation; PSL's `crypto_agility/src/hybrid.rs` integration flagged for additional cryptographer review per ADR-0006. |
| `tokio`                | yes (async runtime internals) | Most-deployed async runtime in the ecosystem. |
| `serde` / `serde_json` | yes (specialization, deserialization fast paths) | RustCrypto-adjacent maturity; widely audited. |
| `zeroize`              | yes (compiler barrier intrinsics) | Specifically designed to defeat compiler dead-store elimination on secret-key zeroing. |

**Posture:** PSL accepts `unsafe` in well-audited cryptographic and
runtime crates. New transitive deps with significant `unsafe` are
flagged at PR review and require justification.

[`cargo-geiger`](https://github.com/rust-secure-code/cargo-geiger)

## Lint configuration

`Cargo.toml` does not currently set `unsafe_code = "forbid"` at the
workspace level. **This is a hardening gap** and is queued for
follow-up:

```toml
# Proposed for v0.1.x → v0.2 hardening pass:
[workspace.lints.rust]
unsafe_code = "forbid"
```

The legacy crate (`legacy/rust_runner/`) would need
`#![allow(unsafe_code)]` at the crate root since it has the one
diagnostic-binary `unsafe` block; that's an acceptable per-crate
allowance because the legacy crate is frozen anyway.

Until the lint lands, the `unsafe` inventory above is maintained by
hand and re-verified by CI grep (planned).

## Tracked: sled migration

The sequencer's backing storage (`psl-sequencer` + `psl-consensus`)
uses [`sled` 0.34.7](https://docs.rs/sled). sled went on hiatus
upstream and the 0.34 line is the last release. This is the source
of three "unmaintained" RUSTSEC advisories currently ignored in
[`/deny.toml`](../deny.toml):

- **RUSTSEC-2025-0057** (`fxhash` unmaintained) — via sled's
  internal hashing.
- **RUSTSEC-2024-0384** (`instant` unmaintained) — via sled's older
  `parking_lot 0.11`.
- **RUSTSEC-2024-0436** (`paste` unmaintained) — via
  `pqcrypto-mldsa` (separate from sled, but same "unmaintained-not-
  exploitable" class).

None of these are exploit-grade vulnerabilities. They are
**unmaintained-status warnings** — the upstream crates are no
longer accepting changes, not actively buggy. Per the cargo-deny
ignore rationale (see `/deny.toml`), each ignore has a written
justification that an auditor can verify.

**v0.2 plan:** migrate sequencer state storage off sled to a
maintained alternative (`rust-rocksdb` is the obvious candidate;
`redb` is a pure-Rust option with single-writer semantics). When
the migration lands, the three RUSTSEC ignores plus the
bitflags=1.3.2 skip in `/deny.toml` can be removed. Tracked in the
follow-up task list; ADR-worth-writing first to lock the
alternative selection.

## Other safety considerations

### Panic-path audit

See [`UNWRAP_AUDIT.md`](UNWRAP_AUDIT.md). 40 production-path
`unwrap()` / `expect()` hits, all in two audited categories:
- 35 lock-poison (`Mutex::lock().unwrap()`) — programming-bug-class
  events; panicking is the correct response per the audit rationale.
- 3 documented structurally-impossible-overflow `expect` (with
  inline justification; for example `count.checked_add(1).expect("thermo length overflow")`
  where `count` is bounded by a `Vec` length already < `isize::MAX`).
- 0 genuine bugs.

### Integer arithmetic

The ternary VM forward kernel uses `checked_add` / `checked_mul`
throughout. Per [`UNWRAP_AUDIT.md`](UNWRAP_AUDIT.md) Pattern 2,
checked-arithmetic failures return `Result`, never panic on
attacker-controllable input.

### Floating point

**Zero floating-point arithmetic on the verifier path** by design
(this is the load-bearing property that makes
dispute-by-re-execution work). Enforced by code review: any PR that
introduces `f32` / `f64` in a verifier-path crate is reviewed
against ADR-0001 / ADR-0006.

### Secret-key handling

Private keys (`SigningKey` instances + the ML-DSA SecretKey bytes
buffer in `HybridSigner`) are wrapped in `ZeroizeOnDrop` so memory
is wiped on scope exit. This is the standard `zeroize` crate
pattern; the `Zeroize` derive on `HybridSigner` (with
`#[zeroize(skip)]` on the public-key cache) ensures the secret
bytes are wiped without the compiler optimizing the wipe away.

## What this document does NOT cover

- Side-channel resistance — see
  [`SECURITY_REVIEW.md`](SECURITY_REVIEW.md) § 10
  ("Side-channel resistance"). Constant-time signing in
  `ed25519-dalek` and `pqcrypto-mldsa` is the dependency-level
  guarantee; PSL's wrappers don't introduce timing-variable code.
- Concurrency safety beyond `unsafe` audit — Rust's type system is
  the primary guarantee. The few sites that touch shared state use
  `Mutex` / `RwLock` (see `UNWRAP_AUDIT.md` Pattern 1).
- WASM / sandbox escape — PSL doesn't ship a WASM runtime in the
  v0.1.0 production path. The legacy WASM-primitive workflow
  (`primitives/`) is frozen per ADR-0001.
